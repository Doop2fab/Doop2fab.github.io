Hunt the Wumpus

This is a program that reads in a file and generates a dungeon for the player to walk through. Your objective is to find the wumpus and shoot him, but not walk into the same room as him or run out of arrows. Also, avoid the traps! When you arrive in a room, you decide whether to move to an adjacent room or to shoot into an adjacent room. There are hints to what may lie beyond the room you are in, so watch out!

Goal: In order to more completely understand how objects work within Java, I worked towards completing a project that reads in a file and generates a dungeon with random layouts.

Challenges: Creating a program that is easy to understand, yet complex and interesting was a problem that I prioritized putting plenty of thought into. Because this project was moreso a lesson in objects, however, I decided to forgo writing any UI. Another interesting problem I faced was reading in a file and creating objects from it, and propagating the dungeon full of dangerous traps.

Solutions: Because this program is supposed to be entertaining for the user, I decided to put my own spin on Gregory Yob's idea (Gregory Yob, "Hunt the Wumpus", Creative Computing, volume 1, 1976, pages 247-250. See http://www.atariarchives.org/bcc1/showpage.php?page=247). This program displays the information about the current room to the user by printing the room number, its description, and the list of adjacent rooms. It also checks if any of the adjacent rooms have spiders, a pit, or the Wumpus and prints a smell/hear message as appropriate. (If adjacent to two pits, it only prints one smelly message, and likewise for spiders.) It also displays the number of remaining arrows the player has. When the player walks into a room with a hazard, they lose. However, if they shoot an arrow into the room with the Wumpus in it, they win! The goal is to shoot into the room with the Wumpus, but not actually walk into its den.

Key Takeaways: I finished this program with a more total understanding of how objects work in Java. Writing this program also demonstrated to me correct ways to not only make a piece of software more entertaining for a user/customer but also how to write code in a more understandable and concise way. Indeed, the value of readable yet concise code was shown to be invaluable.
